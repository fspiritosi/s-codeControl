# COD-391 — Soporte múltiples OC por Factura

**Fecha de inicio:** 2026-04-16
**Estado:** Diseño completado

---

## 1. Análisis

### 1.1 Problema

El flujo actual de alta de facturas de compra solo permite asociar **una única Orden de Compra (OC)** a una factura mediante la FK directa `purchase_invoices.purchase_order_id`. El negocio necesita poder:

1. Crear una factura y vincularla a **múltiples OC del mismo proveedor** (N:1).
2. Cargar en la factura las líneas de **todas las OC seleccionadas** mergeadas en una única grilla.
3. Editar cantidades por línea (el usuario puede facturar menos que lo pendiente).
4. Al confirmar la factura, actualizar `invoiced_qty` y `invoicing_status` de **cada OC afectada**:
   - `FULLY_INVOICED` si todas las líneas de esa OC quedaron totalmente facturadas.
   - `PARTIALLY_INVOICED` si solo algunas o si se redujeron cantidades.
   - `NOT_INVOICED` si no se tocó.
5. Si más tarde se carga **otra factura sobre las mismas OC**, el selector solo debe ofrecer líneas con `pending_qty > 0` (cantidades restantes).
6. Seguir permitiendo N facturas a 1 OC (ya funciona).

En la base ya existe la piedra angular: `purchase_invoice_lines.purchase_order_line_id` (opcional). Cada línea de factura ya puede apuntar a una línea específica de una OC, y distintas líneas de la misma factura pueden apuntar a OCs distintas. Lo que falta es:

- **Esquema**: relajar la semántica de `purchase_invoices.purchase_order_id` (que hoy asume 1 OC).
- **UI**: reemplazar el selector single-select de OC por un multi-select que acumule líneas.
- **Actions**: refactorizar `createPurchaseInvoice` y `confirmPurchaseInvoice` para operar sobre **el conjunto de OCs derivado de las líneas**, no sobre el campo escalar.
- **Visualización**: en el detalle de factura mostrar todas las OC asociadas; en el detalle de OC (ya está) listar todas las facturas que la tocan parcial o totalmente.

Observaciones clave:

- `purchase_invoice_lines.purchase_order_line_id` **ya es opcional y apunta a la línea de OC**, no a la OC. Por lo tanto, agrupando por `purchase_order_line.order_id` obtenemos el conjunto de OCs que una factura cubre sin tocar el esquema.
- `purchase_orders.purchase_invoices` es una relación 1:N (una OC ve varias facturas), pero la reciprocidad `purchase_invoice.purchase_order` **única** está hoy modelada como FK escalar. Esa FK hoy duplica información que ya podemos derivar de las líneas.
- `purchase_order_installments.purchase_invoice_id` ata cada cuota a una factura puntual (para pagos). No cambia con este feature: cada cuota sigue siendo de una sola factura, aunque una factura cubra varias OC — hoy no está en uso activo en el flujo de alta.

#### 1.2.4 Action `confirmPurchaseInvoice` (list/actions.server.ts)

```ts
export async function confirmPurchaseInvoice(id: string) {
  await prisma.purchase_invoices.update({
    where: { id, status: 'DRAFT' }, data: { status: 'CONFIRMED' },
  });

  const invoice = await prisma.purchase_invoices.findUnique({
    where: { id }, include: { lines: true },
  });

  if (invoice?.purchase_order_id) {                        // <-- solo si hay FK escalar
    const poLines = await prisma.purchase_order_lines.findMany({
      where: { order_id: invoice.purchase_order_id },
    });

    for (const invLine of invoice.lines) {
      if (invLine.purchase_order_line_id) {
        await prisma.purchase_order_lines.update({
          where: { id: invLine.purchase_order_line_id },
          data: { invoiced_qty: { increment: Number(invLine.quantity) } },
        });
      }
    }

    const updatedLines = await prisma.purchase_order_lines.findMany({
      where: { order_id: invoice.purchase_order_id },
    });
    const allInvoiced = updatedLines.every((l) => Number(l.invoiced_qty) >= Number(l.quantity));
    const someInvoiced = updatedLines.some((l) => Number(l.invoiced_qty) > 0);

    await prisma.purchase_orders.update({
      where: { id: invoice.purchase_order_id },
      data: {
        invoicing_status: allInvoiced ? 'FULLY_INVOICED' : someInvoiced ? 'PARTIALLY_INVOICED' : 'NOT_INVOICED',
      },
    });
  }
  ...
}
```

Limitaciones:

- El guard `if (invoice?.purchase_order_id)` cortocircuita la actualización del `invoicing_status` **de todas las OC afectadas** si la factura se creó con `purchase_order_id = null` (caso multi-OC).
- El recomputo final de `invoicing_status` toma `order_id: invoice.purchase_order_id`, por lo que **solo actualiza esa única OC**, dejando las otras sin recalcular.
- El incremento de `invoiced_qty` por línea funciona correctamente incluso si son de OCs distintas (itera `invoice.lines.purchase_order_line_id`). Solo el guard y el recálculo de `invoicing_status` están atados a la FK escalar.

#### 1.2.5 Queries de OC para el selector

Ubicación: `src/modules/purchasing/features/purchase-orders/list/actions.server.ts`.

```ts
export async function getOrdersForInvoicing(supplierId: string) {
  const { companyId } = await getActionContext();
  if (!companyId) return [];

  const orders = await prisma.purchase_orders.findMany({
    where: {
      company_id: companyId,
      supplier_id: supplierId,
      status: { in: ['APPROVED', 'PARTIALLY_RECEIVED', 'COMPLETED'] },
      invoicing_status: { in: ['NOT_INVOICED', 'PARTIALLY_INVOICED'] },
    },
    select: { id: true, full_number: true, total: true, issue_date: true, status: true, invoicing_status: true },
    orderBy: { created_at: 'desc' },
  });

  return orders.map((o) => ({ ...o, total: Number(o.total) }));
}

export async function getPurchaseOrderLinesForInvoicing(orderId: string) {
  const lines = await prisma.purchase_order_lines.findMany({
    where: { order_id: orderId },
    include: { product: { select: { id: true, code: true, name: true } } },
  });

  return lines
    .map((l) => ({
      ...l,
      quantity: Number(l.quantity),
      invoiced_qty: Number(l.invoiced_qty),
      pending_qty: Number(l.quantity) - Number(l.invoiced_qty),
      unit_cost: Number(l.unit_cost),
      vat_rate: Number(l.vat_rate),
      vat_amount: Number(l.vat_amount),
      subtotal: Number(l.subtotal),
      total: Number(l.total),
    }))
    .filter((l) => l.pending_qty > 0);
}
```

Ambas se reutilizarán casi tal cual. La única adaptación necesaria será **traer las líneas pendientes de varias OC en una sola llamada** (o llamar a `getPurchaseOrderLinesForInvoicing` N veces, una por OC, en paralelo).

### 1.5 Restricciones y reglas

Reglas CLAUDE.md aplicables:

1. **Migraciones solo ADD** — no se pueden dropear columnas ni FKs. `purchase_invoices.purchase_order_id` queda para siempre como campo opcional.
2. **No cross-module imports** — el form ya importa de `purchase-orders` vía `actions.server`. Ese patrón ya se rompe en la base actual (también lo hace `ReceivingNoteForm`). Se mantiene como está; la alternativa de mover `getOrdersForInvoicing` a shared es fuera de scope.
3. **Server actions por feature** — `createPurchaseInvoice`, `confirmPurchaseInvoice` siguen en `invoices/list/actions.server.ts`.
4. **No se pierden datos en DB** — ver §1.6 riesgo M1.
5. **Commits según reglas** — este cambio toca >5 archivos y >100 líneas (form + actions + schema + migration + views). Al terminar de probar, se hace el commit sin esperar al usuario (regla Git CLAUDE.md).

#### 1.5.1 Opciones de diseño

##### Opción A — Tabla pivote `purchase_invoice_purchase_orders`

```prisma
model purchase_invoice_purchase_orders {
  invoice_id String @db.Uuid
  order_id   String @db.Uuid

  invoice purchase_invoices @relation(fields: [invoice_id], references: [id], onDelete: Cascade)
  order   purchase_orders   @relation(fields: [order_id], references: [id], onDelete: Cascade)

  @@id([invoice_id, order_id])
  @@index([order_id])
}
```

Pros:
- Relación N:N explícita y navegable desde Prisma (`invoice.orders`, `order.invoices`).
- Permite asociar una OC a la factura **incluso si no hay líneas** (caso teórico — en nuestro negocio no aplica porque el usuario arrastra líneas al elegir OC).
- Queries simples: `invoice.include({ orders: true })`.

Contras:
- Nueva tabla + migración add-only.
- Redundancia: la relación ya se puede derivar de `invoice.lines.purchase_order_line.order_id`. Mantener pivote consistente exige escritura transaccional adicional en `createPurchaseInvoice` y en cualquier edición futura de líneas.
- `purchase_invoices.purchase_order_id` queda "zombie": deja de reflejar la realidad cuando hay >1 OC; los consumidores deben ignorarlo (o rellenarlo solo cuando hay exactamente 1).
- Necesita regenerar tipos Prisma y `database.types.ts`.

##### Opción B — Relación implícita por líneas (recomendada)

Sin cambios en el esquema. Se derivan las OCs asociadas a una factura haciendo distinct sobre `invoice.lines[].purchase_order_line.order_id`.

```ts
const invoice = await prisma.purchase_invoices.findUnique({
  where: { id },
  include: {
    lines: {
      include: {
        purchase_order_line: {
          select: { id: true, order_id: true, order: { select: { id: true, full_number: true, invoicing_status: true } } },
        },
      },
    },
  },
});

const linkedOrderIds = Array.from(
  new Set(
    invoice.lines
      .map((l) => l.purchase_order_line?.order_id)
      .filter(Boolean) as string[],
  ),
);
```

Semántica de `purchase_invoices.purchase_order_id`:

- **Se mantiene como campo legacy opcional**. Al crear una factura multi-OC, se puede:
  - (a) dejar en `null` siempre que haya ≥1 línea con `purchase_order_line_id` (derivamos todo por líneas), o
  - (b) popularlo **solo si la factura está atada a una única OC** (retrocompatible con facturas históricas). Recomiendo (b) por UX: si una factura tiene exactamente 1 OC, los listados y filtros que aún usan `purchase_order_id` siguen andando.

Pros:
- **Cero cambios de esquema** → sin migración, sin regenerar tipos, riesgo mínimo.
- Consistencia "gratis": no hay riesgo de que pivote y líneas queden desincronizados.
- Retrocompatibilidad total: facturas históricas con `purchase_order_id` siguen funcionando; el detalle pasa a unificar "OC vía FK" + "OCs vía líneas".
- Se aprovecha infraestructura existente (`purchase_order_line_id` ya es opcional y ya se llena en el flujo actual).
- Se alinea con el principio ADD-only sin tener que agregar estructura nueva (no añadir es también una forma de ADD-only).

Contras:
- No se puede asociar una OC a la factura "sin líneas" (en nuestro caso no lo necesitamos — el usuario siempre carga líneas al elegir OC).
- Query de listado de OCs asociadas requiere un `findMany` extra o un `groupBy distinct` en vez de un include directo (mínimo impacto).
- El detalle de OC (`order.purchase_invoices`) hoy usa `purchase_orders.purchase_invoices` (via FK escalar); deja de funcionar para facturas multi-OC donde `purchase_order_id = null`. Hay que reemplazarlo por una query que derive las facturas de `order.lines[].purchase_invoice_lines[].invoice`.

##### Recomendación: **Opción B**

Razones:

1. **Alineamiento con la regla `ADD-only` y la preferencia del usuario** de evitar reset/migraciones no triviales: la Opción B no requiere migración.
2. **Menor superficie de cambio**: el schema ya expresa perfectamente N:N via `purchase_order_lines ↔ purchase_invoice_lines` (línea a línea). Agregar un pivote a nivel cabecera duplica información.
3. **Consistencia automática**: evitamos escribir lógica para mantener pivote y líneas sincronizados en cada edición o borrado. Si borramos una línea, su OC deja de estar "automáticamente" asociada sin código adicional.
4. **Retrocompatibilidad**: `purchase_order_id` se vuelve un "hint legacy" que podemos seguir rellenando cuando hay exactamente 1 OC. Nada se rompe en datos ya creados.
5. **Detalle de OC**: el cambio de lectura (derivar facturas desde líneas) es simple y además **arregla un bug latente**: hoy si alguien creara una factura con `purchase_order_id = null` pero con líneas apuntando a esa OC, el detalle de la OC no la listaría. Con Opción B esto queda correcto.
6. **Opción A es fácilmente reversible a futuro**: si algún día necesitamos asociar una OC a una factura sin líneas, agregamos la tabla pivote sin borrar nada. No cerramos puertas.

### 1.6 Riesgos identificados

#### R1 — Duplicados en `purchase_order_id`

Si mantenemos `purchase_order_id` con la semántica "única OC", tenemos que decidir cuándo llenarlo:

- **Estrategia propuesta**: llenar `purchase_order_id` **solo si la factura está atada a exactamente 1 OC distinta** (derivado de las líneas al momento del `create`). Si hay 2 o más OCs distintas → `null`.
- **Datos históricos**: todas las facturas existentes tenían a lo sumo 1 OC; `purchase_order_id` puede o no estar — la lógica de detalle debe hacer un UNION (usar FK si existe, fallback a derivar por líneas).

#### R2 — Confirmación parcial: recálculo de `invoicing_status` de múltiples OCs

Hoy `confirmPurchaseInvoice` solo recalcula el `invoicing_status` de `invoice.purchase_order_id`. En multi-OC hay que recalcular **cada OC distinta presente en las líneas**.

Edge case: el `for` que hace `increment: Number(invLine.quantity)` puede cruzar la cota `invoiced_qty > quantity` si la UI permite editar hacia arriba. **Mitigación**: validar en el server que para cada línea `invoice.quantity <= (po_line.quantity - po_line.invoiced_qty)` antes de incrementar, o clampear. Sin validación, se pueden inflar `invoiced_qty` erróneamente y dejar OCs como `FULLY_INVOICED` sin serlo realmente.

#### R3 — Race condition en confirmaciones simultáneas

Dos usuarios confirmando a la vez dos facturas que tocan la misma OC pueden generar lecturas-stale al recomputar `invoicing_status`. El código actual **ya tiene este riesgo** (no lo introduce multi-OC). Mitigación out-of-scope: envolver en `prisma.$transaction` con `SERIALIZABLE` o usar `upsert + calculate`. Documentar pero no resolver en esta tarjeta.

#### R4 — UX de la grilla con múltiples OCs

Si el usuario agrega 3 OCs con 10 líneas cada una, la grilla tendrá 30 filas. Problemas:

- Agrupación visual: el usuario debe saber qué línea vino de qué OC.
- Deselección de OC: si el usuario saca una OC del multi-select, ¿qué pasa con sus líneas ya cargadas? Propuesta: removerlas automáticamente (filtrando por `purchase_order_line_id` resuelto a OC = la removida).
- Líneas "manuales" (sin `purchase_order_line_id`) — el usuario puede agregar líneas sueltas (hoy lo permite). Si hace eso y luego deselecciona TODAS las OCs, esas líneas manuales deben preservarse.

Mitigación:

- Mostrar en la grilla una columna `OC` (read-only, pequeña) con el `full_number` cuando la línea tenga `purchase_order_line_id`.
- `handleOrderToggle(orderId, checked)`:
  - **Si check**: append líneas pendientes de esa OC al `fields` (sin reemplazar las existentes).
  - **Si uncheck**: filtrar y eliminar todas las líneas con `purchase_order_line_id ∈ lines_de_esa_OC`.

#### R5 — Re-carga de líneas pendientes actualizadas entre selecciones

Si el usuario selecciona OC-1 (trae 5 items pendientes), y mientras tanto otro usuario confirma una factura sobre OC-1, al seleccionar OC-2 y volver a mirar OC-1 los pendientes pueden haber cambiado. **Mitigación**: refrescar líneas pendientes al mostrar la lista, y en el server validar nuevamente el `pending_qty` en el momento del `create` (ver R2).

#### R6 — Validación de proveedor único entre OCs

Todas las OCs seleccionadas deben ser del mismo proveedor (ya lo garantiza `getOrdersForInvoicing(supplierId)`), pero si el usuario cambia el proveedor después de seleccionar OCs, hay que limpiar la selección y las líneas derivadas (igual que hoy con single). Este caso ya lo cubre el `useEffect([watchedSupplier])` actual; hay que preservar el patrón.

#### R7 — Detalle de OC que ya dependía de `purchase_orders.purchase_invoices` (FK directa)

`PurchaseOrderDetail.tsx` y `getPurchaseOrderById` incluyen `purchase_invoices` vía relación de FK. En Opción B, esto sigue funcionando para facturas single-OC pero **pierde** las facturas multi-OC donde `purchase_order_id = null`. Cambio requerido:

```ts
// antes
purchase_invoices: { select: { id: true, full_number: true, status: true, total: true } },

// después (derivar de líneas)
lines: {
  include: {
    purchase_invoice_lines: {
      include: { invoice: { select: { id: true, full_number: true, status: true, total: true } } },
    },
  },
},
// + post-proceso en JS: set de invoices únicas agrupadas de todas las lines
```

O alternativamente una query separada `getInvoicesByOrderId(orderId)` que haga:

```ts
const invoices = await prisma.purchase_invoices.findMany({
  where: {
    company_id: companyId,
    OR: [
      { purchase_order_id: orderId },                                        // legacy/single
      { lines: { some: { purchase_order_line: { order_id: orderId } } } },   // multi
    ],
  },
  distinct: ['id'],
  select: { id: true, full_number: true, status: true, total: true },
});
```

La segunda opción es más limpia y centraliza el "UNION" en un único lugar.

#### R8 — Listado de facturas: columna OC

La columna "OCs" (si se agrega) ahora podría ser N valores. Impacto en performance si se incluye en el paginated: habría que hacer un include de `lines.purchase_order_line.order` por cada fila. **Mitigación**: no agregar la columna en el listado (se ve en el detalle). O calcular con un `groupBy` en una segunda query. Para este ticket: postergar la columna al detalle.

#### R9 — `installments` compartidos

Si una factura cubre OC-A (con cuotas 1..3) y OC-B (con cuotas 1..2), y el usuario asocia la factura a "todas las cuotas pendientes", puede terminar asignando cuotas de OCs distintas a una misma factura. El schema lo permite (`purchase_order_installments.purchase_invoice_id` apunta a cualquier factura). No bloqueante, pero debe quedar documentado: este ticket **no** automatiza la asignación de cuotas — sigue siendo manual (si se usa).

#### R10 — Regresión en tests y flujos dependientes

- Recepción desde factura (`ReceivingNoteForm` + `getInvoiceLinesForReceiving`): itera sobre `invoice.lines`, no importa cuántas OCs haya detrás. **No regresiona**.
- Confirmación de nota de crédito (`original_invoice_id`): orthogonal. **No regresiona**.
- Listado de facturas con filtro `receiving_status`: orthogonal. **No regresiona**.
- PDF de OC: lee `order.lines.invoiced_qty`, indirectamente actualizado por `confirmPurchaseInvoice`. **Debe funcionar igual** siempre que el nuevo `confirmPurchaseInvoice` actualice `invoiced_qty` línea a línea (ya lo hace hoy).

## 3. Diseño

### 3.1 Arquitectura general (Opción B — Derivación por líneas, sin migración)

El diseño se apoya en lo que ya existe en el esquema (`purchase_invoice_lines.purchase_order_line_id: String?`). La "lista de OCs" de una factura se **deriva** en server haciendo `distinct` sobre `lines[].purchase_order_line.order_id`. El campo `purchase_invoices.purchase_order_id` se conserva por retrocompatibilidad como **OC primaria**, y se popula solo cuando la factura está atada a exactamente 1 OC distinta.

#### Flujo completo (end-to-end)

```
[Usuario elige proveedor]
        │
        ▼
getOrdersForInvoicing(supplierId)                       ← sin cambios
        │   retorna OCs facturables
        ▼
[Usuario tildea N OCs en MultiSelectCombobox]
        │
        ▼
handleOrdersChange(newIds)
  - Calcula addedIds / removedIds (diff incremental)
  - addedIds → getPurchaseOrderLinesForInvoicingBulk(addedIds)   ← NUEVA
      → append(...) al fieldArray, persiste order_id + order_full_number por línea
  - removedIds → filter del fieldArray por purchase_order_line_id ∈ OC-removida
      (usa Map<line_id → order_id> en useRef)
        │
        ▼
[Usuario edita qty por línea y hace submit]
        │
        ▼
createPurchaseInvoice({ ..., purchase_order_ids: string[], lines: [...] })
  - Deriva primaryOrderId = (ids.length === 1 ? ids[0] : null)
  - Inserta invoice con purchase_order_id = primaryOrderId
  - Inserta lines con purchase_order_line_id por cada una
        │
        ▼
[Usuario confirma factura desde el listado]
        │
        ▼
confirmPurchaseInvoice(id)
  - Lee invoice.lines (include purchase_order_line { id, order_id, quantity, invoiced_qty })
  - Por cada línea con po_line_id: increment clampeado (Math.min(qty, pending))
  - affectedOrderIds = distinct(lines[].purchase_order_line.order_id)
  - Por cada orderId: relee líneas → recalcula invoicing_status
  - Todo dentro de prisma.$transaction([...])
        │
        ▼
[Detalle de factura]
  - Query agrega lines.purchase_order_line.order.{id, full_number}
  - UI: uniqueBy(id) badges (merge invoice.purchase_order legacy + derivadas)

[Detalle de OC]
  - getPurchaseOrderById hace merge del include legacy + getInvoicesByOrderId(id)
  - getInvoicesByOrderId hace UNION (OR entre FK escalar + lines.some.po_line.order_id)
```

Principios:

- **Zero migration**: sin tabla pivote, sin regenerar `database.types.ts`, sin `npm run gentypes`.
- **Single source of truth**: las líneas son la única verdad sobre la relación factura↔OC. `purchase_order_id` es un *hint* legacy reconstruido al momento del insert.
- **Atomicidad**: `confirmPurchaseInvoice` envuelve todos los updates en `$transaction`.
- **Idempotencia defensiva**: clamp `Math.min(invLine.quantity, deltaMax)` evita inflar `invoiced_qty` por encima de `quantity`.

### 3.2 Modelo de datos

Sin cambios en el esquema. Repaso de las columnas relevantes:

```prisma
// YA EXISTE — no se toca
model purchase_invoices {
  purchase_order_id   String?  @db.Uuid   // legacy, semántica "OC primaria"
  ...
  lines               purchase_invoice_lines[]
}

model purchase_invoice_lines {
  purchase_order_line_id String?  @db.Uuid  // puede apuntar a líneas de OCs distintas
  purchase_order_line    purchase_order_lines?  @relation(...)
  ...
}

model purchase_order_lines {
  order_id     String   @db.Uuid
  quantity     Decimal  @db.Decimal(12,3)
  invoiced_qty Decimal  @default(0) @db.Decimal(12,3)
  ...
}
```

**Convención de uso de `purchase_order_id` en escritura:**

| Situación | `purchase_order_id` |
|---|---|
| 0 OCs seleccionadas (factura 100% manual) | `null` |
| 1 OC seleccionada | `= orderIds[0]` |
| ≥ 2 OCs seleccionadas | `null` (multi-OC puro) |

**Convención de lectura en detalle de OC:**

```ts
export interface LineWithPoLine {
  purchase_order_line_id?: string | null;
}

/**
 * Dado un array de líneas de factura (cada una con purchase_order_line_id opcional)
 * y un índice line_id → order_id, retorna el set único de order_ids distintos
 * (filtrando null/undefined). Pura, sin I/O.
 */
export function deriveOrderIdsFromLines(
  lines: LineWithPoLine[],
  poLineToOrderId: Map<string, string>
): string[] {
  const ids = new Set<string>();
  for (const line of lines) {
    const poLineId = line.purchase_order_line_id;
    if (!poLineId) continue;
    const orderId = poLineToOrderId.get(poLineId);
    if (orderId) ids.add(orderId);
  }
  return Array.from(ids);
}

/**
 * Retorna el único order_id si el array tiene exactamente 1, null en cualquier otro caso.
 * Usado para popular purchase_invoices.purchase_order_id (legacy) solo cuando la
 * factura está atada a una única OC.
 */
export function pickPrimaryOrderId(orderIds: string[]): string | null {
  return orderIds.length === 1 ? orderIds[0] : null;
}
```

#### 3.3.2 Query bulk de líneas pendientes — `purchase-orders/list/actions.server.ts`

```ts
/**
 * Retorna todas las líneas pendientes (pending_qty > 0) de múltiples OCs en una
 * única query. Cada línea incluye order_id y order_full_number para agrupación visual.
 * Retorna un array plano (no agrupado): la UI decide cómo mostrar.

export async function getPurchaseOrderLinesForInvoicingBulk(

): Promise<Array<{
  id: string;
  order_id: string;
  order_full_number: string;
  product: { id: string; code: string; name: string } | null;
  description: string;
  quantity: number;
  invoiced_qty: number;
  pending_qty: number;
  unit_cost: number;
  vat_rate: number;
}>> {

  const lines = await prisma.purchase_order_lines.findMany({
    where: { order_id: { in: orderIds } },
    include: {
      product: { select: { id: true, code: true, name: true } },
      order: { select: { full_number: true } },
    },
  });

  return lines
    .map((l) => ({
      id: l.id,
      order_id: l.order_id,
      order_full_number: l.order.full_number,
      product: l.product,
      description: l.description,
```

#### 3.3.3 Query UNION de facturas por OC — `purchase-orders/list/actions.server.ts`

```ts
/**
 * Retorna las facturas vinculadas a una OC, uniendo:
 *   a) FK legacy: purchase_invoices.purchase_order_id = orderId
 *   b) Derivado: facturas con alguna línea cuyo purchase_order_line.order_id = orderId
 * Deduplica por invoice.id. Usa el ctx de empresa actual.
 */
export async function getInvoicesByOrderId(
  orderId: string
): Promise<Array<{
  id: string;
  full_number: string;
  status: string;
  total: number;
}>> {
  const { companyId } = await getActionContext();
  if (!companyId) return [];

  const invoices = await prisma.purchase_invoices.findMany({
    where: {
      company_id: companyId,
      OR: [
        { purchase_order_id: orderId },
        {
          lines: {
            some: { purchase_order_line: { order_id: orderId } },
          },
        },
      ],
    },
    select: {
      id: true,
      full_number: true,
      status: true,
      total: true,
    },
    distinct: ['id'],
    orderBy: { created_at: 'desc' },
  });

  return invoices.map((inv) => ({
    ...inv,
    total: Number(inv.total),
  }));
}
```

#### 3.3.6 `confirmPurchaseInvoice` refactorizado — `invoices/list/actions.server.ts`

```ts
export async function confirmPurchaseInvoice(id: string): Promise<{ error: string | null }> {
  try {
    // 1) Transicionar DRAFT → CONFIRMED (idempotente si se repite)
    await prisma.purchase_invoices.update({
      where: { id, status: 'DRAFT' },
      data: { status: 'CONFIRMED' },
    });

    // 2) Releer factura con líneas + po_line (para clamp y derivación de OCs)
    const invoice = await prisma.purchase_invoices.findUnique({
      where: { id },
      include: {
        lines: {
          include: {
            purchase_order_line: {
              select: { id: true, order_id: true, quantity: true, invoiced_qty: true },
            },
          },
        },
      },
    });
    if (!invoice) return { error: 'Factura no encontrada' };

    // 3) Calcular affectedOrderIds (puede estar vacío si la factura es 100% manual)
    const affectedOrderIds = Array.from(
      new Set(
        invoice.lines
          .map((l) => l.purchase_order_line?.order_id)
          .filter((x): x is string => !!x)
      )
    );

    // 4) Armar los updates de invoiced_qty con clamp defensivo
    const qtyUpdates: Array<{ id: string; delta: number }> = [];
    for (const invLine of invoice.lines) {
      const poLine = invLine.purchase_order_line;
      if (!poLine) continue;
      const poLineQty = Number(poLine.quantity);
      const poLineInvoiced = Number(poLine.invoiced_qty);
      const pending = poLineQty - poLineInvoiced;      // puede ser 0 o negativo si ya se saturó
      const wanted = Number(invLine.quantity);
      const delta = Math.max(0, Math.min(wanted, pending));  // clamp [0, pending]
      if (delta < wanted) {
        console.warn(
          `[confirmPurchaseInvoice] clamp invoice=${id} po_line=${poLine.id} wanted=${wanted} applied=${delta}`
        );
      }
      if (delta > 0) qtyUpdates.push({ id: poLine.id, delta });
    }

    // 5) Ejecutar todos los updates de invoiced_qty en una transacción
    if (qtyUpdates.length > 0) {
      await prisma.$transaction(
        qtyUpdates.map((u) =>
          prisma.purchase_order_lines.update({
            where: { id: u.id },
            data: { invoiced_qty: { increment: u.delta } },
          })
        )
      );
    }

    // 6) Por cada OC afectada, releer sus líneas y recalcular invoicing_status
    if (affectedOrderIds.length > 0) {
      const allLines = await prisma.purchase_order_lines.findMany({
        where: { order_id: { in: affectedOrderIds } },
        select: { order_id: true, quantity: true, invoiced_qty: true },
      });
      const byOrder = new Map<string, typeof allLines>();
      for (const l of allLines) {
        const bucket = byOrder.get(l.order_id) ?? [];
        bucket.push(l);
        byOrder.set(l.order_id, bucket);
      }
      await prisma.$transaction(
        affectedOrderIds.map((orderId) => {
          const lines = byOrder.get(orderId) ?? [];
          const allInvoiced = lines.length > 0 &&
            lines.every((l) => Number(l.invoiced_qty) >= Number(l.quantity));
          const someInvoiced = lines.some((l) => Number(l.invoiced_qty) > 0);
          const newStatus: 'FULLY_INVOICED' | 'PARTIALLY_INVOICED' | 'NOT_INVOICED' =
            allInvoiced ? 'FULLY_INVOICED' : someInvoiced ? 'PARTIALLY_INVOICED' : 'NOT_INVOICED';
          return prisma.purchase_orders.update({
            where: { id: orderId },
            data: { invoicing_status: newStatus },
          });
        })
      );
    }

    revalidatePath('/dashboard/purchasing');
    return { error: null };
  } catch (error) {
    console.error('Error confirming invoice:', error);
    return { error: String(error) };
  }
}
```

Decisiones clave:
- **Guard eliminado**: ya no depende de `invoice.purchase_order_id`; recorre todas las OCs distintas derivadas de `lines`.
- **Clamp**: `delta = max(0, min(wanted, pending))`. Si `wanted > pending`, se aplica solo lo disponible y se loguea warning.
- **Atomicidad**: todos los updates de `invoiced_qty` en un `$transaction` y todos los de `invoicing_status` en otro. Deliberadamente dos transacciones: la primera deja `invoiced_qty` canónico antes de recalcular estados; leer entre ambas garantiza que vemos los valores recién escritos.
- **Idempotencia parcial**: si alguien llama dos veces sobre la misma factura CONFIRMED, el primer `update` con `status: 'DRAFT'` fallará (P2025) porque ya no está en DRAFT — lo captura el try/catch y retorna error. Esto es consistente con el comportamiento actual.

```tsx
const handleOrdersChange = async (newIds: string[]) => {
  const oldIds = (form.getValues('purchase_order_ids') as string[]) || [];
  const addedIds = newIds.filter((id) => !oldIds.includes(id));
  const removedIds = oldIds.filter((id) => !newIds.includes(id));

  // 1) Remover líneas de OCs deseleccionadas (preservar manuales y de otras OCs)
  if (removedIds.length > 0) {
    const currentLines = form.getValues('lines') as LineField[];
    const keep = currentLines.filter((l) => {
      if (!l.purchase_order_line_id) return true;                // manual
      const ord = l.order_id ?? poLineToOrderRef.current.get(l.purchase_order_line_id);
      return ord ? !removedIds.includes(ord) : true;
    });
    replace(keep.length > 0 ? keep : [
      { product_id: '', description: '', quantity: 1, unit_cost: 0, vat_rate: 21 },

    // Limpiar el índice de las OCs removidas
    for (const [lineId, ord] of Array.from(poLineToOrderRef.current.entries())) {
      if (removedIds.includes(ord)) poLineToOrderRef.current.delete(lineId);

  // 2) Agregar líneas de OCs nuevas (append, no replace)
  if (addedIds.length > 0) {
```

```tsx
        point_of_sale: values.point_of_sale,
        number: values.number,
        issue_date: values.issue_date,
        due_date: values.due_date,
        cae: values.cae,
        notes: values.notes,
        purchase_order_ids: values.purchase_order_ids || [],
        lines,
      });
      if (result.error) throw new Error(result.error);
      router.push('/dashboard/purchasing?tab=invoices');
      router.refresh();
    },
    { loading: 'Creando factura...', success: 'Factura creada', error: (e) => e?.message || 'Error' }
  );
};
```

### 3.7 Consideraciones técnicas

#### 3.7.1 Edge cases y manejo esperado

| # | Escenario | Comportamiento esperado |
|---|---|---|
| E1 | Factura sin OC (100% líneas manuales) | `purchase_order_ids = []`, `purchase_order_id = null`. `confirmPurchaseInvoice` no toca ninguna OC. |
| E2 | Factura con 1 OC (flujo legacy) | `purchase_order_ids.length === 1` → `purchase_order_id = orderIds[0]`. Retrocompat total con reportes que todavía lean la FK escalar. |
| E3 | Factura con N OCs (N ≥ 2) | `purchase_order_id = null`, `lines[].purchase_order_line_id` tiene IDs de distintas OCs. Al confirmar, todas las OCs del set afectado se recalculan. |
| E4 | `quantity` editada hacia arriba (> pending) | Clamp en server: `delta = min(wanted, pending)`. Se escribe el delta clampeado, la columna `purchase_invoice_lines.quantity` queda tal cual el usuario la cargó (registro del comprobante), pero `purchase_order_lines.invoiced_qty` nunca supera `quantity`. Log warning. |
| E5 | `quantity` editada hacia abajo | No hay clamp ni ajuste: se incrementa solo el `delta = quantity` real. La OC queda `PARTIALLY_INVOICED` salvo que todas las líneas totalicen. |
| E6 | Cambio de proveedor después de seleccionar OCs | `useEffect([watchedSupplier])` resetea `purchase_order_ids = []` y filtra líneas con `purchase_order_line_id`. Las líneas manuales se preservan. |
| E7 | Deselección de 1 de N OCs | Filtra solo las líneas con `purchase_order_line_id` mapeado a esa OC (vía `poLineToOrderRef`). Preserva líneas de otras OCs y manuales. |
| E8 | Deselección de todas las OCs | Quedan solo líneas manuales; si no hay, inyectar placeholder para respetar `min(1)`. |
| E9 | Factura multi-OC → recepción (ReceivingNoteForm) | Sin cambios: el form de recepción itera `invoice.lines`, no le importa cuántas OCs haya detrás. |
| E10 | Dos facturas distintas sobre misma OC (N:1 ya soportado) | Cada confirmación incrementa `invoiced_qty` en las líneas tocadas y recalcula `invoicing_status` de la OC. |
| E11 | Factura confirmada → `getInvoicesByOrderId` | Aparece tanto para OCs vía FK como derivadas. El detalle de OC la lista independientemente de si tenía `purchase_order_id` seteado. |
| E12 | OC sin líneas (degenerado) | `affectedOrderIds` no la incluye porque no hay líneas; no se recalcula. Consistente con el comportamiento actual. |
| E13 | `purchase_order_ids` duplicado en payload | `Array.from(new Set(...))` dedup en `createPurchaseInvoice`. |

#### 3.7.2 Validaciones

- **Server**: `createPurchaseInvoice` no valida que todas las OCs pertenezcan al mismo proveedor (la UI lo garantiza vía `getOrdersForInvoicing(supplierId)`). **No se agrega validación extra** porque aumenta costo sin beneficio claro; si se desea, se puede añadir un `findMany({ where: { id: { in: orderIds }, supplier_id: data.supplier_id }, select: { id: true } })` y comparar `length === orderIds.length`. Se documenta como mejora futura.
- **Clamp**: ver E4 arriba.
- **Idempotencia de confirm**: el guard `where: { id, status: 'DRAFT' }` garantiza que solo transiciona una vez. Llamadas repetidas retornan error (comportamiento actual preservado).

#### 3.7.3 Performance

- `getPurchaseOrderLinesForInvoicingBulk` hace **una sola query** con `IN` (no N queries en loop). Para N OCs con M líneas cada una es O(N*M) filas en memoria; filtrado `pending_qty > 0` en JS es despreciable.
- `confirmPurchaseInvoice`: 2 `$transaction`s de updates y 2 `findMany` agregados. Sigue siendo O(L) donde L = líneas de la factura. Sin hot path relevante.
- `getInvoicesByOrderId`: un único `findMany` con `OR` y `distinct`. Postgres hace plan óptimo por el índice `@@index([purchase_order_id])` (rama a) y escaneo de `purchase_invoice_lines` por `purchase_order_line.order_id` (rama b — no indexada explícitamente pero la relación ya es selectiva por proveedor).

#### 3.7.4 Patrones seguidos

- **Server actions por feature**: `getPurchaseOrderLinesForInvoicingBulk`, `getInvoicesByOrderId` van en `purchase-orders/list/actions.server.ts`. `createPurchaseInvoice` y `confirmPurchaseInvoice` permanecen en `invoices/list/actions.server.ts`.
- **No cross-module imports**: el form sigue importando de `purchase-orders/list/actions.server.ts` (mismo módulo `purchasing`, submódulos distintos — patrón ya usado por `ReceivingNoteForm`).
- **Helpers puros en `shared`**: `deriveOrderIdsFromLines`, `pickPrimaryOrderId` van en `src/modules/purchasing/shared/invoice-orders.ts`.
- **RHF + Zod**: el form tiene >5 campos y array dinámico → se mantiene RHF (no se aplica la regla `useState+Zod`).
- **ADD-only**: sin migraciones, sin dropear campos. `purchase_order_id` queda como hint legacy.
- **Atomicidad**: `prisma.$transaction` para grupos de updates relacionados en `confirmPurchaseInvoice`.

#### 3.7.5 Limitaciones conocidas (documentar, no resolver en esta tarjeta)

- **Race condition R3** (dos usuarios confirmando facturas que tocan la misma OC simultáneamente): el patrón de "read-modify-write" de `invoicing_status` no está serializado globalmente. Mitigación futura: `SERIALIZABLE` isolation o lock por OC. Ya existía en el código actual; multi-OC no lo empeora significativamente.
- **`installments`**: este ticket no modifica el flujo de cuotas. Una factura multi-OC puede cubrir cuotas de varias OCs (el schema lo permite — `purchase_order_installments.purchase_invoice_id` apunta a una sola factura, pero cada cuota sigue siendo de una sola OC). Se deja fuera de scope.
