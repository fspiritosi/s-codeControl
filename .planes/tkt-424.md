# tkt-424 — Calendario de vencimientos de VTV en el dashboard

**Fecha de inicio:** 2026-07-08
**Estado:** Implementación completada

## 1. Análisis

### 1.1 Problema

El cliente **Transporte SP** (reporta Gonzalo Peralta) gestiona hoy los vencimientos de la **VTV (Verificación Técnica Vehicular)** de su flota con un Excel externo y tiene problemas de control. Quiere reemplazar ese Excel por una funcionalidad dentro de la plataforma:

- Un **calendario de vencimientos de VTV** en el dashboard que muestre automáticamente las VTV próximas a vencer.
- **Estados por color** en el calendario:
  - **Rojo/naranja:** VTV próxima a vencer y todavía SIN orden solicitada.
  - **Verde:** VTV con orden ya solicitada a Transporte (lista para ir a verificar).
  - **Sin color / otro:** VTV ya realizada, para distinguirla fácil.
- Poder **marcar** dentro del sistema cuando una VTV próxima a vencer ya tiene la orden solicitada (cambia de color en el calendario).
- Poder **cancelar, mover o reasignar** una VTV del calendario (modificar el turno / reorganizar la fecha).
- **Notificación de urgencia** cuando el vencimiento esté próximo (ej: falta 1 semana).

Objetivo: centralizar la gestión, dejar de depender del Excel y tener control visual y ordenado de vencimientos, órdenes solicitadas y VTV realizadas.

### 1.2 Contexto actual

**a) Cómo se modela la VTV hoy**

La VTV **no es una entidad propia**: es un **tipo de documento de vehículo**. El modelo relevante:

- Tabla `document_types` (`database.types.ts:1871`): catálogo de tipos de documento. Columnas clave: `name`, `applies` (enum `document_applies`: `Persona` | `Equipos` | `Empresa`), `mandatory`, `explired` (indica si el documento vence / tiene fecha de vencimiento), `is_it_montlhy`, `special` (documento condicional), `conditions` (Json[]), `is_active`, `company_id` (NULL = tipo global de sistema; con valor = tipo propio de la empresa).
- Tabla `documents_equipment` (`database.types.ts:2090`): el documento cargado para un vehículo concreto. Columnas clave: `id`, `applies` (FK → `vehicles.id`, o sea a qué vehículo pertenece), `id_document_types` (FK → `document_types.id`), `validity` (**la fecha de vencimiento, tipo String `YYYY-MM-DD`** — NO se llama `expiration_date`), `state` (enum `state`: incluye `pendiente`, `vencido`, `rechazado`, etc.), `document_path`, `is_active`, `period`.
- Tabla `vehicles` (`database.types.ts`): flota. Columnas útiles para mostrar en el calendario: `domain` (patente), `intern_number` (interno), `brand`, `model`, `type_of_vehicle`, `company_id`, `is_active`.

**Importante:** La "VTV" tendría que identificarse por un `document_types` cuyo `name` sea "VTV" (o similar). Hoy NO hay ninguna referencia a "VTV" en el código; se identifica por nombre del tipo de documento cargado por cada empresa (ver riesgos y preguntas abiertas). El vencimiento de una VTV concreta = fila de `documents_equipment` con ese `id_document_types` y su `validity`.

**b) Dashboard de vencimientos existente (reutilizable)**

- `src/modules/dashboard/features/overview/actions.server.ts` — Server action `getDashboardCounts()`. Ya calcula documentos por vencer / vencidos de empleados y equipos usando Prisma. Patrón: `getActionContext()` para obtener `companyId`, filtros sobre `documents_equipment` con `validity` en rango `{ gte: today, lte: nextMonth }`, scope por `vehicle.company_id`, `unstable_cache` con tag por empresa (`revalidate: 60`). Distingue tipos condicionales (`special=true`, evaluados en JS con `checkDocumentAppliesToEquipment`) de no condicionales (count directo en SQL). **Este archivo es el molde exacto de cómo consultar VTV próximas a vencer.**
- `src/modules/dashboard/features/tables/components/VPendingDocumentTable.tsx` y `EPendingDocumentTable.tsx` — Tablas de documentos pendientes de vehículos/empleados en el dashboard. `VPendingDocumentTable` lee `pendingDocuments.vehicles` del store Zustand (`useLoggedUserStore`) y lo pinta con `ExpiredDataTable` + `ExpiredColums` (de `@/modules/documents/shared/columns/ExpiredColumns`). **OJO:** este componente hace un cross-module import a `@/modules/documents/...` (deuda técnica ya presente, no replicar).
- `src/modules/dashboard/features/overview/components/DashboardComponent.tsx` + `CardsGrid.tsx` — Composición actual del dashboard (grilla de cards por módulo). Aquí se insertaría el card/sección del calendario de VTV.

**c) Notificaciones (mecanismo actual — reutilizable)**

Sistema de notificaciones ya montado y funcionando (Fase 6):

- `src/shared/services/notifications.ts` — `createNotification({ typeCode, companyId, metadata, dedupeKey, recipientProfileIds?, excludeProfileIds? })`. Resuelve destinatarios por `required_permission_code` del `notification_type` (owner + user_roles + fallback legacy), renderiza `title/description/link` desde templates `{{var}}`, crea `notifications` + `notification_recipients` en transacción. **Idempotente vía `dedupeKey`** (UNIQUE por `company_id + dedupe_key`).
- `src/shared/actions/notifications.ts` — Wrappers server action para el consumo desde UI (listar, marcar leído, dismiss, contador de no leídas).
- Tabla `notification_types` (catálogo): `code`, `title_template`, `description_template`, `category` (enum `notification_categories`, ej. `vencimiento`, `advertencia`), `required_permission_code`, `link_template`, `is_active`. Se siembran por migración SQL (ver `supabase/migrations/20260505131018_seed_permissions_and_system_roles.sql:218`, ej. `documents.expiring_soon` con link `/dashboard/document?status=expiring`).
- **Disparo automático:** `src/app/api/cron/notifications/route.ts` — Endpoint HTTP (`GET`/`POST`, auth por header `x-cron-secret` == env `CRON_SECRET`) **disparado por Supabase `pg_cron` diariamente**. Itera empresas activas y genera notificaciones agregadas: documentos vencidos, documentos por vencer en 5 días (`countExpiringSoonDocuments` filtra `documents_equipment.validity` entre hoy y +5, `state notIn ['vencido','rechazado']`), OC pendientes, OP pendientes. Usa `dedupeKey` con fecha (`documents.expiring_soon:${todayStr}`) para no duplicar en corridas del mismo día. **Este es el patrón exacto para la "notificación de urgencia de VTV": agregar un bloque más en este cron (o un tipo nuevo `vtv.expiring_soon`).**

**d) Calendario (componentes existentes — reutilizables)**

- `src/shared/components/ui/calendar.tsx` — Componente shadcn/ui basado en **`react-day-picker`** (`DayPicker`), con locale `es` (date-fns/locale). Permite `classNames` y `components` custom → se puede colorear días con modificadores (`modifiers` / `modifiersClassNames` de react-day-picker) para pintar rojo/verde/neutro por estado de VTV.
- `src/shared/components/ui/enhanced-datepicket.tsx` — DatePicker con selects de mes/año, útil para el modal de "mover/reasignar turno".
- `src/shared/components/ui/FormItemDatePicker.tsx` — DatePicker integrado a formularios.
- **No existe hoy** un componente de "calendario de eventos/agenda" (tipo grilla mensual con items por día). Para el calendario de VTV probablemente haya que construir una vista custom encima de `react-day-picker` (modifiers por día) o una grilla propia; `react-day-picker` ya está disponible (no requiere dependencia nueva).

**e) Estado nuevo requerido (NO existe hoy)**

Los conceptos **"orden solicitada", "VTV realizada", "turno", "cancelar/mover/reasignar"** NO existen en el modelo actual. `documents_equipment.state` (enum `state`) es sobre el ciclo del *documento* (pendiente/presentado/vencido/rechazado), no sobre el *trámite de turno de VTV*. Hará falta un artefacto nuevo (ver §1.6 y §1.7 — decisión de diseño, no se define acá): probablemente una tabla nueva de "turno/orden de VTV" con estados (`pendiente` / `orden_solicitada` / `realizada` / `cancelada`), fecha de turno, y vínculo al vehículo y/o al `documents_equipment` de la VTV.

**f) Multi-tenant**

Todas las queries son company-scoped. El `companyId` se obtiene server-side vía `getActionContext()` (que lee la cookie `actualComp`), como se ve en `overview/actions.server.ts` (`const { companyId } = await getActionContext()`). El cron, en cambio, itera **todas** las empresas activas (`prisma.company.findMany({ where: { is_active: true } })`) porque corre sin sesión.

### 1.5 Restricciones y reglas (CLAUDE.md)

- **Arquitectura modular:** la feature va en `src/modules/{módulo}/features/{feature}/` con `actions.server.ts` (queries+mutations), `components/`, `index.ts` (barrel).
- **`src/app/` es SOLO routing** — la página del dashboard es un wrapper fino que importa el componente del módulo.
- **No cross-module imports** — un módulo no importa de otro; lo compartido va a `src/shared/`. (Ojo: `VPendingDocumentTable` ya viola esto importando de `@/modules/documents`; no replicar ese antipatrón.)
- **Server actions por feature** — cada feature con su `actions.server.ts` (`'use server'`), usando `getActionContext()`/`actualComp`.
- **Multi-tenant** — toda query scoped por `companyId` (cookie `actualComp`); el cron itera empresas.
- **Migraciones solo ADD** (memoria del proyecto): nunca perder datos, evitar `db reset`. Crear tabla nueva y columnas nullable; no alterar destructivamente. Flujo: `npm run create-migration -- <nombre>` → probar local → `npm run push-migrations`.
- **`database.types.ts` es autogenerado** — no editar a mano; regenerar con `npm run gentypes`.
- **Serverless (incidente registrado):** **NO usar SSE en Vercel** (dispara timeouts). Para tiempo real / disparos periódicos usar **polling o cron** (pg_cron → endpoint), tal como ya hace `api/cron/notifications`.

### 1.6 Riesgos identificados

1. **Identificar la VTV entre los `document_types`.** No hay marca de "VTV" en el modelo; depende del `name` que cada empresa le puso al tipo de documento (podría ser "VTV", "Verificación Técnica", "RTO", etc.), y `document_types` puede ser global (`company_id = null`) o por empresa. Riesgo de matchear mal o de que Transporte SP no lo tenga cargado con el nombre esperado. Necesita una forma robusta de marcarlo (flag/columna nueva en `document_types`, o config por empresa).
2. **Estado nuevo (turno/orden) que no existe.** Hay que modelar un ciclo de vida (pendiente → orden_solicitada → realizada / cancelada) + fecha de turno + reasignación. Decidir si es tabla nueva vinculada a `documents_equipment` o a `vehicles`, y cómo se relaciona con el `validity` del documento cuando la VTV se realiza (¿actualiza `validity`? ¿crea nuevo documento?).
3. **Notificación automática en serverless.** La "urgencia (falta 1 semana)" debe engancharse en el cron diario existente (`api/cron/notifications`) o un tipo nuevo `vtv.expiring_soon` — **nunca SSE**. Cuidar `dedupeKey` para no spamear.
4. **Migración sobre producción sin perder datos.** Tabla y columnas nuevas deben ser aditivas (nullable / con default), sin `db reset`. Regenerar tipos tras el push.
5. **Coloreado del calendario.** `react-day-picker` colorea por día, pero puede haber **múltiples VTV el mismo día** con estados distintos; hay que definir cómo se representa (badge por vehículo dentro del día vs. color del día). Puede requerir vista custom en vez de solo modifiers.
6. **Cross-module imports.** Ubicar la feature de forma que no necesite importar de otros módulos; extraer a `shared/` lo que sea común (columnas, tipos de documento VTV).
7. **Performance / cache.** Si se usa `unstable_cache` como en overview, invalidar el tag de empresa al marcar orden/mover turno para que el calendario refleje el cambio.

### 1.7 Decisiones de producto (confirmadas por el usuario)

1. **Identificación de la VTV → flag nueva en `document_types`.** Se agrega una columna booleana (p.ej. `is_vtv`, nullable/default false) a `document_types` para marcar el/los tipo(s) que representan la VTV, en vez de depender del `name`. Migración **solo-ADD**. (Definir en diseño: si la marca un admin desde el ABM de tipos de documento y si aplica a tipos globales o por empresa.)
2. **El turno/orden de VTV es una entidad NUEVA propia del sistema.** Tabla nueva con su ciclo de vida (p.ej. `pendiente` → `orden_solicitada` → `realizada` / `cancelada`) + **fecha de turno propia**, vinculada al vehículo y al `documents_equipment` de la VTV. El calendario muestra el **turno agendado** (fecha propia del sistema), **separado del vencimiento real** del documento.
3. **"Mover / cancelar / reasignar" afecta SOLO el turno**, nunca el `validity` del documento. Reprogramar cambia la fecha del turno; cancelar cambia su estado. El vencimiento real del documento no se toca desde el calendario.
4. **Al marcar "VTV realizada": el sistema NO hace nada con el documento.** El documento sigue actualizándose por su flujo normal de documentos (carga/renovación habitual). El calendario **solo marca el turno como `realizada`** y cambia el color.
5. **Notificación de urgencia → ambos canales: in-app + email.** Se usa el sistema de notificaciones existente (nuevo `notification_type`, p.ej. `vtv.expiring_soon`) enganchado en el **cron diario** (`api/cron/notifications`) para el in-app, y además envío por email (nodemailer/Resend ya configurado). **Nunca SSE.** Umbral base "falta ~1 semana".

### 1.8 Pendientes menores para diseño (no bloquean)

- Umbral de la notificación: fijo "1 semana" vs configurable, y a qué `required_permission_code` / rol se notifica.
- Ubicación exacta en el dashboard (card en overview vs sección/ruta propia) y qué roles pueden operar los turnos (¿Guest/Auditor excluidos?).
- Coloreado cuando hay múltiples turnos de VTV el mismo día (color del día vs badges por vehículo dentro del día).
- ¿Un vehículo puede acumular varios turnos históricos (uno por vencimiento anual)? — probable que sí; definir unicidad del turno "activo".

## 3. Diseño

### 3.1 Arquitectura de la solución

**Ubicación del feature:** `src/modules/dashboard/features/vtv-calendar/`

Se ubica en el módulo `dashboard` porque el entregable es un "calendario en el dashboard" y consume la superficie del overview. Toca `vehicles` y `documents_equipment` **solo vía Prisma** (que es `shared`), por lo que **no genera cross-module imports**. Alternativa descartada: `equipment/features/vtv/` (más coherente por dominio, pero obligaría a duplicar la integración con el dashboard y la ruta vive bajo `/dashboard`).

```
src/modules/dashboard/features/vtv-calendar/
├── actions.server.ts          # 'use server' — queries + mutations (company-scoped)
├── types.ts                   # tipos TS del feature (VtvCalendarItem, VtvAppointmentStatus...)
├── components/
│   ├── VtvView.tsx            # server component raíz (fetch + layout de la página /dashboard/vtv)
│   ├── VtvCalendar.tsx        # client — react-day-picker con modifiers por estado
│   ├── VtvDayPanel.tsx        # client — lista de turnos del día seleccionado (badges por vehículo)
│   ├── VtvAppointmentDialog.tsx # client — gestión de un turno (orden/realizada/cancelar/reprogramar)
│   └── VtvCalendarSkeleton.tsx
└── index.ts                   # barrel: export { VtvView }
```

**Flujo de datos (server → client):**
1. `src/app/dashboard/vtv/page.tsx` (wrapper fino) renderiza `<VtvView/>` dentro de `<Suspense>`.
2. `VtvView` (server) llama a `ensureVtvAppointments()` (autogenera turnos faltantes de VTV por vencer) y luego `getVtvAppointments()` → devuelve `VtvCalendarItem[]` ya scoped por empresa.
3. Pasa los items como props al `VtvCalendar` (client), que agrupa por fecha y computa `modifiers`.
4. Las mutaciones (marcar orden / realizada / cancelar / reprogramar) son **server actions** invocadas desde `VtvAppointmentDialog`; tras cada una se hace `router.refresh()` en el cliente (sin SSE) y la action invalida el tag de cache de la empresa.

**Scope por empresa:** todas las queries/mutations obtienen `companyId` server-side vía `getActionContext()` (cookie `actualComp`), calcando `overview/actions.server.ts`. El cron itera todas las empresas activas (sin sesión).

**Reutiliza:** `Calendar` (`src/shared/components/ui/calendar.tsx`), `enhanced-datepicket.tsx` (reprogramar), `getActionContext()`, `prisma`, `unstable_cache`/`revalidateTag`, `createNotification`, `sendEmail`, el cron existente. **Crea:** tabla `vtv_appointments`, columna `document_types.is_vtv`, `notification_type` `vtv.expiring_soon`, el feature completo, la ruta `/dashboard/vtv` y el link de sidebar.

### 3.2 Modelos de datos

**Migración** (`npm run create-migration -- vtv_appointments`) — **solo-ADD**, sin `db reset`. El proyecto **no usa RLS en tablas de negocio** (ej. `expenses` no tiene policies; el scoping es a nivel app vía Prisma), así que la tabla nueva **no lleva RLS**. Usa `CREATE TYPE ... ENUM` como el resto (ej. `expense_status`).

```sql
-- 1. Flag para identificar el/los tipo(s) de documento que representan la VTV.
--    Aditivo, nullable con default. Lo marca un admin desde el ABM de tipos
--    de documento (aplica tanto a tipos globales company_id=NULL como por empresa).
ALTER TABLE document_types
  ADD COLUMN is_vtv boolean DEFAULT false;

-- 2. Ciclo de vida del turno/orden de VTV.
CREATE TYPE vtv_appointment_status AS ENUM (
  'pendiente', 'orden_solicitada', 'realizada', 'cancelada'
);

-- 3. Turno de VTV (entidad nueva propia del sistema).
--    appointment_date = fecha del TURNO (propia), SEPARADA del validity real
--    del documento. Mover/cancelar afecta solo esta fila, nunca documents_equipment.
CREATE TABLE vtv_appointments (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  company_id            uuid NOT NULL REFERENCES company(id) ON DELETE CASCADE,
  vehicle_id            uuid NOT NULL REFERENCES vehicles(id) ON DELETE CASCADE,
  document_equipment_id uuid REFERENCES documents_equipment(id) ON DELETE SET NULL,
  appointment_date      date NOT NULL,
  status                vtv_appointment_status NOT NULL DEFAULT 'pendiente',
  notes                 text,
  created_by            text,               -- profile.id que lo generó (o 'system' en autogen)
  is_active             boolean NOT NULL DEFAULT true,  -- soft-delete (patrón del proyecto)
  created_at            timestamptz NOT NULL DEFAULT now(),
  updated_at            timestamptz NOT NULL DEFAULT now()
);

-- Índices
CREATE INDEX idx_vtv_appointments_company_id ON vtv_appointments(company_id);
CREATE INDEX idx_vtv_appointments_vehicle_id ON vtv_appointments(vehicle_id);
CREATE INDEX idx_vtv_appointments_doc_eq_id  ON vtv_appointments(document_equipment_id);
CREATE INDEX idx_vtv_appointments_status     ON vtv_appointments(status);
CREATE INDEX idx_vtv_appointments_date       ON vtv_appointments(appointment_date);

-- Unicidad del turno "activo": un solo turno vigente (pendiente u orden_solicitada)
-- por documento de VTV. Los históricos (realizada/cancelada) quedan libres, de modo
-- que un vehículo acumula un turno por vencimiento anual.
CREATE UNIQUE INDEX uq_vtv_active_per_doc
  ON vtv_appointments(document_equipment_id)
  WHERE status IN ('pendiente', 'orden_solicitada') AND is_active = true;
```

export async function getVtvAppointments(

// crea un turno status='pendiente', appointment_date = validity del documento,

  documentEquipmentId?: string | null;
  appointmentDate: string;       // YYYY-MM-DD
  notes?: string | null;
}): Promise<{ id?: string; error?: string }>

export async function markOrderRequested(appointmentId: string): Promise<ActionResult> // → 'orden_solicitada'
export async function markCompleted(appointmentId: string): Promise<ActionResult>       // → 'realizada' (NO toca el documento)

### 3.7 Consideraciones técnicas

- **Patrones calcados:** query cacheada + `getActionContext()` de `overview/actions.server.ts`; cron con `dedupeKey` fechado de `api/cron/notifications`; `createNotification` sin lanzar (safe-to-fail); migración estilo `expenses` (enum + tabla + índices, sin RLS).
- **Orden de invalidación:** primero la mutación en DB, luego `revalidateTag('vtv-${companyId}')` + `revalidatePath('/dashboard/vtv')` en la action; el cliente hace `router.refresh()`. Sin SSE (incidente registrado en memoria del proyecto).
- **Idempotencia:** cron con `dedupeKey='vtv.expiring_soon:${todayStr}'` (in-app) y el email atado a `!r.skipped` para no reenviar el mismo día. `ensureVtvAppointments` idempotente por el índice único parcial `uq_vtv_active_per_doc`.
- **Migración aditiva:** `ADD COLUMN` nullable + `CREATE TABLE`/`CREATE TYPE` nuevos; nada destructivo; sin `db reset`. `gentypes` después del push.
- **No cross-module imports:** el feature solo usa `prisma`, `shared/*` y componentes UI de `shared`. No importa de `@/modules/documents` ni `@/modules/equipment` (a diferencia del antipatrón de `VPendingDocumentTable`).
- **Trade-offs:** (1) `is_vtv` como flag booleana permite que una empresa marque más de un tipo como VTV; el calendario los une por `is_vtv=true`. (2) El color del día por prioridad puede ocultar visualmente que hay estados mixtos; se compensa con el badge contador y el panel del día. (3) Enviar un email agregado por empresa (no por vehículo) evita spam. (4) Resolver emails por permiso exige refactor menor (exportar `resolveProfilesByPermission`); alternativa sin refactor: query directa de owner + user_roles en el cron (más código duplicado).
