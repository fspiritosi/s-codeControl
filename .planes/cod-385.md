# COD-385 — Bug de documentos duplicados

**Fecha de inicio:** 2026-04-16
**Estado:** Diseno completado

---

## 1. Analisis

### 1.1 Problema
En la vista de documentos (`/dashboard/document`), algunos documentos aparecen duplicados o triplicados en las tablas de empleados y equipos.

### 1.2 Contexto actual

Se identificaron **3 causas raiz** que contribuyen a la duplicacion:

#### CAUSA 1 (PRINCIPAL): `updateDocumentByAppliesAndType` usa `updateMany` sin constraint unico

**Archivo:** `src/modules/documents/features/manage/actions.server.ts` (lineas 110-145)

La funcion `updateDocumentByAppliesAndType` hace un `updateMany` filtrando por `applies` + `id_document_types`. El problema es que **no existe un constraint unico** en la combinacion `(applies, id_document_types)` en la tabla `documents_employees` ni en `documents_equipment`.

```typescript
// Linea 118-127 — manage/actions.server.ts
const data = await prisma.documents_employees.updateMany({
  where: { applies: appliesId, id_document_types: documentTypeId },
  data: updateData as any,
});
// If no existing record was found, create one (upsert logic)
if (data.count === 0) {
  await prisma.documents_employees.create({
    data: { applies: appliesId, id_document_types: documentTypeId, ...updateData } as any,
  });
}
```

El esquema Prisma muestra que el unico constraint unico es `document_path`:

```prisma
// prisma/schema.prisma lineas 509-527
model documents_employees {
  id                String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  id_document_types String?  @db.Uuid
  applies           String?  @db.Uuid
  document_path     String?  @unique(map: "documents_employees_docoment_path_key")
  // ...sin constraint unico en (applies, id_document_types)
}
```

**Consecuencia:** Cuando se crea un empleado, `useEmployeeForm.ts` (linea 220) ejecuta `insertDocumentsEmployees(documentsMissing)` que inserta un row por cada tipo de documento mandatorio. Si el proceso se ejecuta mas de una vez (por ejemplo, por un retry del usuario, o si el empleado ya existia con documentos previos), se crean multiples filas con el mismo `applies` + `id_document_types`. Cuando luego se sube un documento, `updateMany` actualiza **todos** los duplicados, dejandolos todos visibles.

#### CAUSA 2: `uploadDocument` en `utils.ts` — flujo no-mandatorio inserta sin verificar existencia

**Archivo:** `src/shared/lib/utils.ts` (lineas 175-235)

Para documentos **no mandatorios** (`mandatory === false`), la funcion `uploadDocument` siempre hace un INSERT nuevo:

```typescript
// Linea 226-232 — utils.ts
} else {
  const { error } = await insertSingleDocumentEmployee({
    ...dataToUpdate,
    state: 'presentado',
  });
}
```

No hay verificacion de si ya existe un documento del mismo tipo para el mismo recurso. Si el usuario sube el mismo tipo de documento dos veces (por ejemplo, con diferente vencimiento), se crean dos filas.

#### CAUSA 3: MultiResourceDocument — itera creando un INSERT por recurso sin dedup

**Archivo:** `src/modules/documents/features/upload/components/MultiResourceDocument.tsx` (lineas 201-235)

Para documentos multirecurso no mandatorios, se hace un loop insertando por cada recurso:

```typescript
// Linea 222-229
const insertFn = tableName === 'documents_employees' ? insertSingleDocumentEmployee : insertSingleDocumentEquipment;
const { error } = await insertFn({
  ...tableEntries[index],
  state: 'presentado',
  document_path: uploadPath,
});
```

Si el usuario sube el mismo documento multirecurso dos veces (la validacion de storage solo chequea existencia del archivo en el bucket, no en la DB), se crean filas duplicadas en la base de datos.

#### CAUSA 4 (MENOR): useEffect con `router.refresh()` en TabsDocuments (legacy)

**Archivo:** `src/modules/documents/features/list/components/TabsDocuments.tsx` (lineas 51-53)

```typescript
useEffect(() => {
  router.refresh();
}, [actualComp]);
```

Este `router.refresh()` se dispara cada vez que cambia la cookie `actualComp`, lo que re-renderiza el componente. Aunque no causa duplicacion en la DB, puede causar un flash visual de datos si el store de Zustand y los datos del server se superponen temporalmente.

**Nota:** La pagina principal (`src/app/dashboard/document/page.tsx`) usa la nueva arquitectura server-side con `EmployeeDocumentsTabs` y `EquipmentTabs` que usan `getEmployeeDocumentsPaginated` / `getEquipmentDocumentsPaginated`. Estas queries son correctas (no tienen JOINs problematicos, usan Prisma `findMany` con `include` que no genera duplicados a nivel SQL). El problema NO esta en las queries de lectura, sino en la escritura.

### 1.5 Restricciones y reglas

- `database.types.ts` es auto-generado — no editar manualmente
- Migraciones DB deben ser solo ADD (no perder datos)
- `src/app/` es solo routing — no poner logica
- No cross-module imports

### 1.6 Riesgos identificados

1. **Datos existentes duplicados en produccion:** Antes de agregar un constraint unico `(applies, id_document_types)` hay que limpiar los duplicados existentes. Requiere una migracion cuidadosa que:
   - Identifique filas duplicadas (mismo `applies` + `id_document_types`)
   - Conserve la fila mas reciente (o la que tenga `document_path` no null)
   - Elimine las demas
   - Solo entonces agregue el constraint unico

2. **Documentos mensuales:** Para documentos mensuales (`is_it_montlhy: true`), puede ser valido tener multiples filas con el mismo `applies` + `id_document_types` pero diferente `period`. El constraint unico deberia incluir `period` o manejar este caso aparte.

3. **Documentos con `document_path` null vs no-null:** Los documentos "pendientes" (alertas mandatorias) tienen `document_path: null`. Cuando se sube el archivo, se hace `updateMany`. Si hay duplicados pendientes, todos se actualizan. El constraint unico `document_path` impide que dos filas tengan el mismo path, pero como los pendientes tienen `null` (y SQL trata null como distinto de null en unique), se permite la duplicacion.

4. **Race conditions:** Si dos usuarios suben el mismo documento simultaneamente para el mismo empleado/tipo, ambos pueden crear una fila (sin constraint unico para prevenirlo).

5. **Impacto en reportes/conteo:** Los duplicados inflan los conteos de facets y el total de documentos mostrados en la paginacion.

## 3. Diseno

### 3.1 Arquitectura de la solucion

El approach es **deduplicacion defensiva en cada punto de escritura**, sin migracion de DB. Se aplica en 4 capas:

1. **Capa de upsert (`manage/actions.server.ts`):** Reemplazar `updateMany` ciego por logica find-one + update + cleanup de sobrantes.
2. **Capa de insert no-mandatorio (`upload/actions.server.ts`):** Nueva funcion helper `findExistingDocument` que permite a los callers decidir entre insert y update.
3. **Capa de componente multirecurso (`MultiResourceDocument.tsx`):** Usar el helper para hacer upsert por recurso en lugar de insert ciego.
4. **Capa de creacion de alertas (`useEmployeeForm.ts`, `useVehicleForm.ts`):** Filtrar tipos de documento que ya existen antes de hacer `createMany`.

**Regla clave para documentos mensuales:** Una fila es considerada duplicada solo si coincide en `(applies, id_document_types, period)`. Dos filas con distinto `period` son validas y no deben deduplicarse entre si.

### 3.2 Funciones y metodos

#### 3.2.1 NUEVA: `findExistingDocument` — `src/modules/documents/features/upload/actions.server.ts`

Busca un documento existente por la tupla `(applies, id_document_types)` y opcionalmente `period` para mensuales.

```typescript
'use server';

/**
 * Busca un documento existente para la combinacion (applies, id_document_types).
 * Para documentos mensuales, incluye `period` en el filtro.
 * Retorna la fila "ganadora" (prioridad: document_path no null, luego mas reciente).
 * Si no existe, retorna null.
 */
export const findExistingDocument = async (
  tableName: 'documents_employees' | 'documents_equipment',
  appliesId: string,
  documentTypeId: string,
  period?: string | null
): Promise<{ data: { id: string; document_path: string | null } | null; error: string | null }> => {
  try {
    const where: any = {
      applies: appliesId,
      id_document_types: documentTypeId,
    };
    // Solo agregar period al filtro si se proporciona (documento mensual)
    if (period !== undefined && period !== null) {
      where.period = period;
    }

    const findMany = tableName === 'documents_employees'
      ? prisma.documents_employees.findMany
      : prisma.documents_equipment.findMany;

    const rows = await findMany({
      where,
      select: { id: true, document_path: true, created_at: true },
      orderBy: { created_at: 'desc' },
    });

    if (rows.length === 0) return { data: null, error: null };

    // Seleccionar ganadora: prioridad a la que tiene document_path, sino la mas reciente
    const winner = rows.find((r) => r.document_path !== null) ?? rows[0];
    return { data: { id: winner.id, document_path: winner.document_path }, error: null };
  } catch (error) {
    console.error('Error finding existing document:', error);
    return { data: null, error: String(error) };
  }
};
```

**Ubicacion:** Se agrega en `src/modules/documents/features/upload/actions.server.ts` porque es una query de lectura del dominio de documentos, junto a las demas funciones de fetch de ese archivo.

#### 3.2.2 NUEVA: `fetchExistingDocumentTypes` — `src/modules/documents/features/upload/actions.server.ts`

Obtiene los `id_document_types` que ya tienen al menos una fila para un `applies` dado. Se usa en la Fase 4 (guard de alertas mandatorias).

```typescript
'use server';

/**
 * Retorna los id_document_types que ya existen para un applies dado.
 * Usado para filtrar alertas mandatorias antes de insertar.
 */
export const fetchExistingDocumentTypes = async (
  tableName: 'documents_employees' | 'documents_equipment',
  appliesId: string
): Promise<{ data: string[]; error: string | null }> => {
  try {
    const findMany = tableName === 'documents_employees'
      ? prisma.documents_employees.findMany
      : prisma.documents_equipment.findMany;

    const rows = await findMany({
      where: { applies: appliesId },
      select: { id_document_types: true },
    });

    const types = [...new Set(
      rows
        .map((r) => r.id_document_types)
        .filter((t): t is string => t !== null)
    )];

    return { data: types, error: null };
  } catch (error) {
    console.error('Error fetching existing document types:', error);
    return { data: [], error: String(error) };
  }
};
```

#### 3.2.3 MODIFICAR: `updateDocumentByAppliesAndType` — `src/modules/documents/features/manage/actions.server.ts`

Reemplazar el `updateMany` ciego por logica de find-winner + update + cleanup.

**Firma (sin cambios en la interfaz publica):**
```typescript
export const updateDocumentByAppliesAndType = async (
  tableName: 'documents_employees' | 'documents_equipment',
  appliesId: string,
  documentTypeId: string,
  updateData: Record<string, unknown>
): Promise<{ data: any; error: string | null }>
```

**Pseudocodigo de la nueva logica interna:**

```typescript
// 1. Construir filtro base
const where: any = { applies: appliesId, id_document_types: documentTypeId };

// 2. Si updateData incluye period (documento mensual), agregarlo al filtro
if (updateData.period !== undefined && updateData.period !== null) {
  where.period = updateData.period;
}

// 3. findMany con el filtro, ordenado por created_at desc
const rows = await prisma.documents_employees.findMany({
  where,
  select: { id: true, document_path: true, created_at: true },
  orderBy: { created_at: 'desc' },
});

if (rows.length === 0) {
  // 4a. No existe: crear (mantener fallback actual)
  await prisma.documents_employees.create({
    data: { applies: appliesId, id_document_types: documentTypeId, ...updateData } as any,
  });
  return { data: { count: 1 }, error: null };
}

// 4b. Seleccionar ganadora: prioridad a la que tiene document_path, sino la mas reciente (index 0)
const winner = rows.find((r) => r.document_path !== null) ?? rows[0];

// 5. Actualizar solo la ganadora
const data = await prisma.documents_employees.update({
  where: { id: winner.id },
  data: updateData as any,
});

// 6. Eliminar sobrantes (si habia duplicados)
const surplusIds = rows.filter((r) => r.id !== winner.id).map((r) => r.id);
if (surplusIds.length > 0) {
  await prisma.documents_employees.deleteMany({
    where: { id: { in: surplusIds } },
  });
}

return { data, error: null };
```

La misma logica se replica en la rama `documents_equipment`. Para evitar duplicacion de codigo, se puede extraer una funcion interna `upsertAndCleanup` parametrizada por el modelo Prisma, o simplemente duplicar las ~20 lineas en cada rama (mas simple dado que Prisma no tiene un modelo generico).

### 3.3 Consideraciones tecnicas

#### Patrones a seguir
- Todas las funciones nuevas en `upload/actions.server.ts` llevan `'use server'` (ya declarado en el top del archivo).
- Retorno consistente con `{ data, error }` pattern usado en todo el modulo.
- Tipos `Record<string, unknown>` para `updateData` (consistente con las funciones existentes).
- `as any` en Prisma cuando se pasan objetos dinamicos (patron existente en el codebase).

#### Edge cases
1. **Documento mensual sin period:** Si `period` es `undefined`/`null` y el doc type es `is_it_montlhy: true`, el filtro NO incluye `period`, asi que matchea cualquier fila existente. Esto es correcto porque el caller siempre deberia enviar `period` para mensuales; si no lo hace, se comporta como no-mensual.
2. **Race condition entre dos uploads simultaneos:** Sin constraint unico en DB, sigue siendo posible un race condition si dos requests concurrentes hacen el `findExistingDocument` al mismo tiempo y ambas obtienen `null`. La probabilidad es baja y la dedup de la Fase 1 (`updateDocumentByAppliesAndType`) actua como segunda linea de defensa limpiando sobrantes en el siguiente upload.
3. **document_path unique constraint:** Al hacer update de una fila existente con un nuevo `document_path`, si el path anterior era unico, no hay conflicto. Si se intenta asignar un path que ya tiene otra fila, Prisma lanzara un error de unique constraint — esto es correcto y debe propagarse.
4. **Cleanup de sobrantes en `updateDocumentByAppliesAndType`:** El `deleteMany` de IDs sobrantes es seguro porque solo elimina filas que comparten la misma tupla `(applies, id_document_types, period?)`. No elimina filas de otros tipos o periodos.
5. **`updateDocumentsByAppliesArrayAndType` ahora itera en lugar de batch:** Esto agrega N queries en lugar de 1, pero N es tipicamente < 50 (cantidad de recursos seleccionados en multirecurso). El trade-off es aceptable porque garantiza la deduplicacion por recurso individual.
6. **No se necesita importar `insertSingleDocumentEquipment` en `utils.ts` actualmente** porque la rama single-resource + equipment no existe en el flujo actual (solo `insertSingleDocumentEmployee`). Sin embargo, para que la rama `findExistingDocument` + `updateDocumentById` funcione para ambas tablas, la logica del `else` (insert) deberia usar la funcion correcta segun `tableName`. Agregar la logica condicional como se muestra en 3.2.5.
