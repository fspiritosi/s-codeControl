## 1. Analisis

### 1.1 Problema

El filtro de "Recurso" (columna `resource`) en las tablas de documentos de empleados y equipos no funciona. El usuario puede escribir en el input de texto del filtro, pero la tabla no se filtra: siempre muestra los mismos resultados sin importar lo que se escriba.

**Causa raiz:** Las funciones `buildEmployeeDocumentsWhere` y `buildEquipmentDocumentsWhere` en `actions.server.ts` no procesan el filtro `resource` que llega desde la URL. El filtro se envia correctamente como query param (`?resource=valor`), se parsea correctamente en `state.filters.resource`, pero la funcion `buildEmployeeDocumentsWhere` simplemente lo ignora -- no tiene codigo para transformar ese filtro en una clausula `where` de Prisma.

### 1.2 Contexto actual

#### Configuracion del filtro en el DataTable (funciona correctamente)

En `_EmployeeDocumentDataTable.tsx` linea 27:
```typescript
{ columnId: 'resource', title: 'Empleado', type: 'text' },
```

En `_EquipmentDocumentDataTable.tsx` linea 27:
```typescript
{ columnId: 'resource', title: 'Equipo (Dominio)', type: 'text' },
```

Ambos estan en `DEFAULT_VISIBLE_FILTERS` (linea 33), asi que aparecen visibles por defecto. El tipo `'text'` hace que se renderice un input de texto libre. Cuando el usuario escribe algo, el DataTable agrega `resource=valor` a la URL via `onColumnFiltersChange` -> `updateURL`.

#### Parseo de searchParams (funciona correctamente)

En `helpers.ts` linea 42-49, `parseSearchParams` toma todos los params que no son reservados (`page`, `pageSize`, `sortBy`, `sortOrder`, `search`) y los mete en `state.filters`. Entonces `resource=valor` se convierte en `state.filters.resource = ['valor']`.

#### Donde falla: buildEmployeeDocumentsWhere (linea 885)

El filtro `resource` **nunca se procesa**. El mismo problema existe en `buildEquipmentDocumentsWhere` (linea 1141).

#### Tambien falta: allocated_to

Hay otro filtro de texto configurado (`allocated_to`, "Afectado a") que tampoco tiene manejo server-side, pero eso es un tema separado.

#### Comparacion con filtros que SI funcionan

- **`state`** (faceted): procesado en linea 911-913 directamente como `baseWhere.state = ...`
- **`mandatory`** (faceted): procesado en linea 918-922 como filtro en la relacion `document_type.mandatory`
- **`documentName`** (text): procesado en linea 926-928 como filtro en la relacion `document_type.name`
- **`validity`** (dateRange): procesado en lineas 932-939

Notar que `documentName` se declara como `type: 'text'` en `FILTER_DEFINITIONS` linea 28, pero su manejo en `buildEmployeeDocumentsWhere` es como filtro exacto (`{ in: values }`) en vez de usar `contains` con `mode: 'insensitive'`. Esto es un bug menor adicional: el filtro de texto de documentName espera un match exacto del nombre, no un substring.

#### Que necesita el filtro de resource

Para empleados, `resource` es un campo **computado** por `formatEmployeeDocuments`: es `"Lastname Firstname"`. No existe como campo en la base de datos. El filtro necesita traducirse a una busqueda en `employee.lastname` y/o `employee.firstname`.

Para equipos, `resource` es `vehicle.domain`. El filtro necesita traducirse a una busqueda en `vehicle.domain`.

#### columnMap (linea 840)

```typescript
const employeeDocColumnMap: Record<string, string> = {
  state: 'state',
  documentName: 'document_type.name',
  mandatory: 'document_type.mandatory',
};
```

No incluye `resource`. Pero ademas, aunque se agregara, no se podria resolver con un simple mapeo porque `resource` para empleados necesita buscar en una **relacion anidada** (`employee.lastname` + `employee.firstname`), no un campo directo.

### 1.4 Dependencias

- El campo `resource` es **computado** despues del query, no es un campo de la DB. Esto implica que el filtro debe traducirse a campos reales de la relacion.
- Para empleados: `resource` = `employee.lastname` + `employee.firstname`
- Para equipos: `resource` = `vehicle.domain`
- El helper `buildTextFiltersWhere` existe en `helpers.ts` (L209-225) pero no se usa en las funciones de documentos. No sirve directamente porque `resource` mapea a campos en relaciones anidadas, no a campos directos.

### 1.5 Restricciones y reglas

- **No cross-module imports**: la solucion debe estar dentro del modulo `documents` o en `shared/`.
- **Server actions en modules**: las queries paginadas estan correctamente en `actions.server.ts`.
- **`database.types.ts` es auto-generado**: no se debe editar.
- La solucion no debe alterar el schema de la base de datos.

### 1.6 Riesgos identificados

1. **Busqueda en campo computado**: Como `resource` es la concatenacion de `lastname + firstname`, un filtro "contains" en ambos campos por separado puede dar resultados ligeramente diferentes a buscar en el string concatenado. Ejemplo: buscar "ez Ju" (final de apellido + inicio de nombre) no matchearia buscando en cada campo individualmente. Este es un tradeoff aceptable -- el comportamiento correcto para la mayoria de los casos es buscar en ambos campos con OR.

2. **Bug adicional en `documentName`**: El filtro de `documentName` esta declarado como `type: 'text'` (texto libre) pero su manejo server-side (L926-928) usa match exacto (`{ in: values }`). Si el usuario escribe un substring, no va a encontrar nada. Esto podria repararse junto con este ticket o en uno separado.

3. **Filtro `allocated_to` tambien falta**: Mismo problema que `resource`, sin manejo server-side. Para empleados es el nombre del contratista (relacion anidada `contractor_employee.contractor.name`), para equipos es el tipo de vehiculo (relacion `type_of_vehicle_rel.name`).

### 1.7 Solucion propuesta (resumen)

Agregar en `buildEmployeeDocumentsWhere` el manejo de `state.filters.resource`:

```typescript
// Empleados: buscar en employee.firstname y employee.lastname
const resourceFilter = state.filters.resource;
if (resourceFilter?.length && resourceFilter[0]) {
  const term = resourceFilter[0];
  const resourceConditions = [
    { employee: { is: { firstname: { contains: term, mode: 'insensitive' } } } },
    { employee: { is: { lastname: { contains: term, mode: 'insensitive' } } } },
  ];
  // Si ya hay un OR (por search), combinar con AND
  if (baseWhere.OR) {
    baseWhere.AND = [...(baseWhere.AND as any[] || []), { OR: resourceConditions }];
  } else {
    baseWhere.OR = resourceConditions;
  }
}
```

Y en `buildEquipmentDocumentsWhere`:

```typescript
// Equipos: buscar en vehicle.domain
const resourceFilter = state.filters.resource;
if (resourceFilter?.length && resourceFilter[0]) {
  const term = resourceFilter[0];
  baseWhere.vehicle = {
    is: {
      ...(baseWhere.vehicle as any)?.is,
      domain: { contains: term, mode: 'insensitive' },
    },
  };
}
```

Impacto: 2 funciones en 1 archivo (`actions.server.ts`), ~20 lineas de codigo.
