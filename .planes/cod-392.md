## 1. Analisis

### 1.1 Problema

Actualmente, el flujo de remitos de recepcion (receiving notes) solo permite vincular una Orden de Compra (OC) como origen. Al seleccionar un proveedor, se cargan las OC pendientes de recibir y el usuario puede elegir una para pre-cargar las lineas del remito.

Falta implementar el flujo paralelo: **Factura de compra -> Remito -> Stock**. En escenarios donde el proveedor no emite OC previa, o donde la factura es el documento principal de referencia, se necesita poder crear un remito vinculado a una factura confirmada y que al confirmar el remito:
- Se ingrese stock al almacen (igual que con OC)
- Se actualice `received_qty` en las lineas de la factura
- Se actualice un estado de recepcion en la factura (nuevo campo)

```typescript
  const { companyId } = await getActionContext();
```

```typescript
export async function getPurchaseOrderLinesForReceiving(orderId: string) {
  const lines = await prisma.purchase_order_lines.findMany({
    where: { order_id: orderId },
    include: { product: { select: { id: true, code: true, name: true, unit_of_measure: true, track_stock: true } } },
  });

  return lines
    .filter((l) => l.product?.track_stock)
    .map((l) => ({
      ...l,
      quantity: Number(l.quantity),
      received_qty: Number(l.received_qty),
      pending_qty: Number(l.quantity) - Number(l.received_qty),
      unit_cost: Number(l.unit_cost),
      vat_rate: Number(l.vat_rate),
    }))
    .filter((l) => l.pending_qty > 0);
}
```

```typescript
export async function confirmReceivingNote(id: string) {
  const { companyId } = await getActionContext();
  if (!companyId) throw new Error('No company selected');

  try {
    const note = await prisma.receiving_notes.findUnique({
      where: { id },
      include: {
        lines: { include: { product: { select: { track_stock: true } } } },
      },
    });

    if (!note || note.status !== 'DRAFT') {
      return { error: 'El remito no esta en estado borrador' };
    }

    const operations: any[] = [
      // 1. Mark as CONFIRMED
      prisma.receiving_notes.update({
        where: { id },
        data: { status: 'CONFIRMED' },
      }),
    ];

    for (const line of note.lines) {
      if (!line.product?.track_stock) continue;

      // 2a. Create stock movement
      operations.push(
        prisma.stock_movements.create({
          data: {
            company_id: companyId,
            warehouse_id: note.warehouse_id,
            product_id: line.product_id,
            type: 'PURCHASE',
            quantity: Number(line.quantity),
            reference_type: 'receiving_note',
            reference_id: note.id,
            date: note.reception_date,
          },
        })
      );

      // 2b. Upsert warehouse stock
      operations.push(
        prisma.warehouse_stocks.upsert({
          where: {
            warehouse_id_product_id: {
              warehouse_id: note.warehouse_id,
              product_id: line.product_id,
            },
          },
          update: {
            quantity: { increment: Number(line.quantity) },
            available_qty: { increment: Number(line.quantity) },
          },
          create: {
            warehouse_id: note.warehouse_id,
            product_id: line.product_id,
            quantity: Number(line.quantity),
            available_qty: Number(line.quantity),
            reserved_qty: 0,
          },
        })
      );

      // 2c. Increment receivedQty on PO line if linked
      if (line.purchase_order_line_id) {
        operations.push(
          prisma.purchase_order_lines.update({
            where: { id: line.purchase_order_line_id },
            data: { received_qty: { increment: Number(line.quantity) } },
          })
        );
      }
    }

    await prisma.$transaction(operations);

    // 3. Update PO status if linked
    if (note.purchase_order_id) {
      await updatePurchaseOrderStatus(note.purchase_order_id);
    }

    revalidatePath('/dashboard/purchasing');
    revalidatePath('/dashboard/warehouse');
    return { error: null };
  } catch (error) {
    console.error('Error confirming receiving note:', error);
    return { error: String(error) };
  }
}
```

```typescript
export async function cancelReceivingNote(id: string) {
  const { companyId } = await getActionContext();
  if (!companyId) throw new Error('No company selected');

  try {
    const note = await prisma.receiving_notes.findUnique({
      where: { id },
      include: {
        lines: { include: { product: { select: { track_stock: true } } } },
      },
    });

    if (!note || note.status !== 'CONFIRMED') {
      return { error: 'Solo se pueden cancelar remitos confirmados' };
    }

    const operations: any[] = [
      prisma.receiving_notes.update({ where: { id }, data: { status: 'CANCELLED' } }),
    ];

    for (const line of note.lines) {
      if (!line.product?.track_stock) continue;

      // Reverse stock movement
      operations.push(
        prisma.stock_movements.create({
          data: {
            company_id: companyId,
            warehouse_id: note.warehouse_id,
            product_id: line.product_id,
            type: 'ADJUSTMENT',
            quantity: -Number(line.quantity),
            reference_type: 'receiving_note_cancellation',
            reference_id: note.id,
            date: new Date(),
          },
        })
      );

      // Decrement stock
      operations.push(
        prisma.warehouse_stocks.update({
          where: {
            warehouse_id_product_id: {
              warehouse_id: note.warehouse_id,
              product_id: line.product_id,
            },
          },
          data: {
            quantity: { decrement: Number(line.quantity) },
            available_qty: { decrement: Number(line.quantity) },
          },
        })
      );

      // Decrement receivedQty on PO line
      if (line.purchase_order_line_id) {
        operations.push(
          prisma.purchase_order_lines.update({
            where: { id: line.purchase_order_line_id },
            data: { received_qty: { decrement: Number(line.quantity) } },
          })
        );
      }
    }

    await prisma.$transaction(operations);

    if (note.purchase_order_id) {
      await updatePurchaseOrderStatus(note.purchase_order_id);
    }

    revalidatePath('/dashboard/purchasing');
    revalidatePath('/dashboard/warehouse');
    return { error: null };
  } catch (error) {
    console.error('Error cancelling receiving note:', error);
    return { error: String(error) };
  }
}
```

```typescript
export async function updatePurchaseOrderStatus(orderId: string) {
  const lines = await prisma.purchase_order_lines.findMany({ where: { order_id: orderId } });

  const allReceived = lines.every((l) => Number(l.received_qty) >= Number(l.quantity));
  const someReceived = lines.some((l) => Number(l.received_qty) > 0);

  let newStatus: 'APPROVED' | 'PARTIALLY_RECEIVED' | 'COMPLETED' = 'APPROVED';
  if (allReceived) newStatus = 'COMPLETED';
  else if (someReceived) newStatus = 'PARTIALLY_RECEIVED';

  await prisma.purchase_orders.update({
    where: { id: orderId },
    data: { status: newStatus },
  });
}
```

```typescript
export async function confirmPurchaseInvoice(id: string) {
  try {
    await prisma.purchase_invoices.update({
      where: { id, status: 'DRAFT' },
      data: { status: 'CONFIRMED' },
    });

    const invoice = await prisma.purchase_invoices.findUnique({
      where: { id },
      include: { lines: true },
    });

    if (invoice?.purchase_order_id) {
      // Increment invoicedQty on linked PO lines
      for (const invLine of invoice.lines) {
        if (invLine.purchase_order_line_id) {
          await prisma.purchase_order_lines.update({
            where: { id: invLine.purchase_order_line_id },
            data: { invoiced_qty: { increment: Number(invLine.quantity) } },
          });
        }
      }

      // Update PO invoicing status
      const updatedLines = await prisma.purchase_order_lines.findMany({
        where: { order_id: invoice.purchase_order_id },
      });
      const allInvoiced = updatedLines.every((l) => Number(l.invoiced_qty) >= Number(l.quantity));
      const someInvoiced = updatedLines.some((l) => Number(l.invoiced_qty) > 0);

      await prisma.purchase_orders.update({
        where: { id: invoice.purchase_order_id },
        data: {
          invoicing_status: allInvoiced ? 'FULLY_INVOICED' : someInvoiced ? 'PARTIALLY_INVOICED' : 'NOT_INVOICED',
        },
      });
    }

    revalidatePath('/dashboard/purchasing');
    return { error: null };
  } catch (error) {
    console.error('Error confirming invoice:', error);
    return { error: String(error) };
  }
}
```

### 1.6 Riesgos identificados

1. **Remitos mixtos (OC + Factura)**: Un remito podria tener `purchase_order_id` Y `purchase_invoice_id` al mismo tiempo? Esto ocurriria si una factura esta vinculada a una OC. Hay que decidir si se permite o si son mutuamente excluyentes. Recomendacion: permitirlo cuando la factura tiene `purchase_order_id` (son parte del mismo flujo). El form deberia forzar uno u otro como seleccion primaria.

2. **Lineas de factura sin producto**: `purchase_invoice_lines.product_id` es nullable (`String?`), pero para recibir stock necesitas un producto con `track_stock`. Las lineas sin producto (servicios, gastos) deben filtrarse en `getInvoiceLinesForReceiving()`, igual que `getOrderLinesForReceiving` filtra por `track_stock`.

3. **Doble conteo de stock**: Si un remito se vincula a una factura que a su vez esta vinculada a una OC, y ambos tienen lineas vinculadas a `purchase_order_lines`, confirmar el remito podria incrementar `received_qty` en PO lines dos veces (una por `purchase_order_line_id` del remito, otra por la logica de invoice). Hay que asegurarse de que solo se incremente una vez. El flujo actual solo incrementa desde el remito, no desde la factura — esto esta OK.

4. **Facturas con notas de credito**: Las facturas tipo NC (NOTA_CREDITO_*) no deberian ser elegibles para recepcion. Filtrar por `voucher_type` que sea FACTURA_* o NOTA_DEBITO_*.

5. **Concurrencia**: Dos remitos creados en paralelo para la misma factura podrian exceder la cantidad total. Se mitiga con la transaccion en `confirmReceivingNote()`, pero no hay validacion de `received_qty <= quantity` a nivel de DB. Considerar agregar CHECK constraint o validacion en la accion.

6. **Nuevo enum vs campo string**: Para `receiving_status` en `purchase_invoices`, se puede usar un enum nuevo (como `purchase_order_invoicing_status`) o reusar la misma logica. Recomendacion: crear enum `purchase_invoice_receiving_status` con valores `NOT_RECEIVED | PARTIALLY_RECEIVED | FULLY_RECEIVED`.

## 3. Diseno

### 3.1 Arquitectura — Extension del flujo existente

El flujo actual es **OC -> Remito -> Stock**. Se agrega un flujo paralelo **Factura -> Remito -> Stock** sin modificar el existente.

```
                 ┌─── OC ──────────┐
                 │  (purchase_orders) │
Proveedor ──►   │                     │──► Remito de Recepcion ──► Stock
                 │ (purchase_invoices) │    (receiving_notes)       (warehouse_stocks)
                 └─── Factura ────┘
```

**Principios de extension:**
- `receiving_notes` ya tiene `purchase_order_id` y `purchase_invoice_id` (ambos opcionales). No se tocan.
- `receiving_note_lines` tiene `purchase_order_line_id` pero NO `purchase_invoice_line_id`. Se agrega.
- `purchase_invoice_lines` no tiene `received_qty`. Se agrega (analogo a `purchase_order_lines.received_qty`).
- `purchase_invoices` no tiene `receiving_status`. Se agrega (analogo a `purchase_orders.status` para recepcion).
- La exclusion mutua OC/Factura se maneja en la UI (al seleccionar uno se limpia el otro), no con constraint de DB.
- El flujo de stock (stock_movements + warehouse_stocks) es identico para ambos origenes; la unica diferencia es que linea de factura o de OC se incrementa.

**Flujo de datos al confirmar un remito vinculado a factura:**
1. `confirmReceivingNote(id)` — lee el remito con sus lineas
2. Transaccion:
   - Marca remito como CONFIRMED
   - Para cada linea con `track_stock`: crea stock_movement + upsert warehouse_stocks
   - Si `line.purchase_order_line_id`: incrementa `purchase_order_lines.received_qty` (existente)
   - Si `line.purchase_invoice_line_id`: incrementa `purchase_invoice_lines.received_qty` (NUEVO)
3. Post-transaccion:
   - Si `note.purchase_order_id`: `updatePurchaseOrderStatus(orderId)` (existente)
   - Si `note.purchase_invoice_id`: `updatePurchaseInvoiceReceivingStatus(invoiceId)` (NUEVO)

```typescript
export async function updatePurchaseInvoiceReceivingStatus(invoiceId: string) {
  // Lee TODAS las lineas de la factura que tienen producto con track_stock
  const lines = await prisma.purchase_invoice_lines.findMany({
    where: { invoice_id: invoiceId },
    include: { product: { select: { track_stock: true } } },
  });

  // Solo considerar lineas con tracking de stock para determinar el estado
  const trackableLines = lines.filter((l) => l.product?.track_stock);

  // Si la factura no tiene lineas con stock trackeable, no cambiar nada
  if (trackableLines.length === 0) return;

  const allReceived = trackableLines.every(
    (l) => Number(l.received_qty) >= Number(l.quantity)
  );
  const someReceived = trackableLines.some(
    (l) => Number(l.received_qty) > 0
  );

  let newStatus: 'NOT_RECEIVED' | 'PARTIALLY_RECEIVED' | 'FULLY_RECEIVED' = 'NOT_RECEIVED';
  if (allReceived) newStatus = 'FULLY_RECEIVED';
  else if (someReceived) newStatus = 'PARTIALLY_RECEIVED';

  await prisma.purchase_invoices.update({
    where: { id: invoiceId },
    data: { receiving_status: newStatus },
  });
}
```

### 3.7 Consideraciones tecnicas

#### 3.7.1 Edge cases

**Remito ya tiene OC y se intenta vincular factura (o viceversa):**
- No es posible desde la UI. Los botones OC/Factura son mutuamente excluyentes. Al seleccionar uno se limpia el otro (`form.setValue('purchase_order_id', '')` y `form.setValue('purchase_invoice_id', '')`).
- En la DB, ambos campos son nullable e independientes. No hay constraint de exclusion mutua. Si alguien crea un remito via API con ambos, `confirmReceivingNote` manejara ambos flujos correctamente (incrementa received_qty en ambas lineas y actualiza ambos estados).

**Factura vinculada a una OC:**
- Una factura puede tener `purchase_order_id` (fue creada desde una OC). Si se crea un remito desde esa factura, se vincula `purchase_invoice_id` y `purchase_invoice_line_id` pero NO `purchase_order_id` ni `purchase_order_line_id`. Esto evita el doble conteo de `received_qty` en PO lines.
- El usuario que quiera recibir mercaderia debe elegir UNO de los dos flujos: OC o Factura. No se mezclan.

**Lineas de factura sin producto:**
- `getInvoiceLinesForReceiving()` filtra por `product.track_stock === true`. Las lineas sin producto (servicios, gastos) o con `track_stock === false` se excluyen automaticamente.

**Facturas tipo NC o Recibo:**
- `getInvoicesForReceiving()` filtra por `voucher_type` que sea FACTURA_* o NOTA_DEBITO_*. Las notas de credito y recibos se excluyen.

**Facturas en estado DRAFT:**
- `getInvoicesForReceiving()` requiere `status: 'CONFIRMED'`. No se puede crear un remito de recepcion desde una factura en borrador.

**Factura ya totalmente recibida:**
- `getInvoicesForReceiving()` filtra por `receiving_status: { in: ['NOT_RECEIVED', 'PARTIALLY_RECEIVED'] }`. Una factura `FULLY_RECEIVED` no aparece en el selector.

**Concurrencia — dos remitos para la misma factura:**
- La transaccion de `confirmReceivingNote` usa `$transaction` lo cual serializa las operaciones. Sin embargo, dos transacciones concurrentes podrian ambas pasar si la suma excede la cantidad de la linea. Esto es el mismo comportamiento que tiene el flujo de OC (no hay CHECK constraint a nivel DB).
- Para la v1, se acepta este riesgo. Si se quiere mitigar en el futuro, se puede agregar una validacion pre-transaccion que verifique `current_received_qty + line.quantity <= invoice_line.quantity`.

**Cancelacion de remito vinculado a factura:**
- `cancelReceivingNote` decrementara `received_qty` en las lineas de factura y recalculara `receiving_status`. Si `received_qty` llega a 0, el estado vuelve a `NOT_RECEIVED`. Esto es identico al flujo de OC.

#### 3.7.2 Validaciones

| Validacion | Donde | Como |
|-----------|-------|------|
| Factura debe estar CONFIRMED | `getInvoicesForReceiving` | where `status: 'CONFIRMED'` |
| Excluir NC/Recibos | `getInvoicesForReceiving` | where `voucher_type: { in: [...] }` |
| Solo productos con track_stock | `getInvoiceLinesForReceiving` | filter `l.product?.track_stock` |
| Solo lineas con pending_qty > 0 | `getInvoiceLinesForReceiving` | filter `pending_qty > 0` |
| OC y Factura mutuamente excluyentes | UI (ReceivingNoteForm) | Al seleccionar uno se limpia el otro |
| `purchase_invoice_line_id` es UUID valido o vacio | Validator | `z.string().uuid().optional().or(z.literal(''))` |
| Lineas sin stock en factura no impiden receiving_status | `updatePurchaseInvoiceReceivingStatus` | Solo evalua lineas con `track_stock` |
